## Theory anchors (Chapter 1)

The chapter names **four properties** and maps them onto APM's **three promises**
([`content/playbook-brief.md`](../playbook-brief.md), [official docs](https://microsoft.github.io/apm/)):

| Property (Ch1 frame) | Promise (APM tagline) | File / feature that implements it |
|---|---|---|
| **Portability** | *Portable by manifest* | `apm.yml` + harness targets + `apm install` |
| **Reproducibility** | (byte-for-byte restore) | `apm.lock.yaml` |
| **Provenance / security** | *Secure by default* | content hashes in `apm.lock.yaml`; install-time scans |
| **Governance** | *Governed by policy* | `apm-policy.yml` |

Every feature note below links back to one of these.

## Grounded snippet — the real `apm init` manifest (v0.23.1)

Generated verbatim by `apm init -y` in the scratch dir (nothing added). This is the exact shape the
author should show as "the empty manifest":

```yaml
# apm.yml -- generated by `apm init -y` on apm-cli 0.23.1 (the empty manifest)
name: apm-ch01
version: 1.0.0
description: APM project for apm-ch01
author: Ravi Sharma
# Which agent platforms to deploy to (uncomment to pin):
# targets:
#   - copilot
#   - claude

dependencies:
  apm: []
  mcp: []
includes: auto
scripts: {}
```

Notes for the author:
- `targets:` is emitted **commented out** — an opt-in, not a default (see the "no default copilot"
  caveat below).
- `dependencies` is split into two typed buckets from the start: `apm:` (primitives) and `mcp:`
  (MCP servers). Both empty on a fresh project.
- `includes: auto` = auto-discover locally-authored primitives in the project.
- `scripts: {}` = the (experimental) `apm run <script>` surface; empty by default.
- `apm init` created **only** `apm.yml` — no lockfile, no policy file, no harness dirs. Those appear
  later in the lifecycle.

## Feature reference

### 1. `apm.yml` — the manifest
- **Implements concept:** **Portability** → *portable by manifest*. One declarative file travels with
  the repo and describes the agent context every developer should get.
- **Usage:** The single human-authored source of truth. You declare project identity
  (`name`/`version`/`description`/`author`), the primitives and MCP servers you depend on
  (`dependencies.apm` / `dependencies.mcp`), which harnesses to deploy to (`targets:`), and scripts.
  Analogy for the reader: `package.json` / `requirements.txt` / `Cargo.toml`, but for agent context.
- **When to use / when not:** Always present in an APM project (created by `apm init`). Edit `apm.yml`
  to *declare intent*; don't hand-edit `apm.lock.yaml` (that's derived). Keep it minimal and pin
  dependency refs so the lockfile is deterministic.
- **Minimal example:** the generated snippet above. A one-dependency illustration (Ch4 territory,
  shown here only to preview pinning):
  ```yaml
  # illustrative only -- pins a public sample package by ref (verify live in Ch4/Ch5)
  dependencies:
    apm:
      - microsoft/apm-sample-package
  ```
  Marked **SKIPPED-needs-network** for Ch1 (no live installs here).

### 2. `apm.lock.yaml` — the lockfile
- **Implements concept:** **Reproducibility** (byte-for-byte) and half of **provenance/security**
  (content hashes).
- **Usage:** The machine-generated resolution of `apm.yml`: **exact versions + content hashes** so a
  fresh clone reproduces the same agent setup byte-for-byte
  ([docs](https://microsoft.github.io/apm/)). Written by resolving dependencies —
  `apm lock` "Resolve dependencies and write `apm.lock.yaml` without deploying files" (CLI help,
  v0.23.1) — and consumed by `apm install --frozen` for a CI-safe, exact restore.
- **When to use / when not:** **Commit it.** Let APM write it; never hand-edit. Use
  `apm install --frozen` in CI to refuse drift ("Refuse to install when `apm.lock.yaml` is missing or
  out of sync with `apm.yml`", CLI help). Full treatment in Ch6.
- **Minimal example:** conceptual for Ch1 — the file is produced by resolution, not authored. Real
  lockfile contents are verified in Ch6.

### 3. `apm-policy.yml` — the policy file
- **Implements concept:** **Governance** → *governed by policy*.
- **Usage:** Org-level governance **enforced at install time, including transitive MCP**, with
  **tighten-only inheritance** flowing **enterprise → org → repo**
  ([docs](https://microsoft.github.io/apm/)). It is **discovered from your project's git remote**
  (v0.23.1 `apm policy status --policy-source` tries the org's `.github`, `.apm`, then `_apm` repos;
  Azure DevOps uses `_apm`) rather than sitting in the project by default. Inspect the effective
  posture with `apm policy status` and `apm policy explain <package>`.
- **When to use / when not:** An **enterprise/producer** concern — most consumers only *experience*
  policy (an install is warned or blocked), they don't author it. In Ch1 it's named, not written.
  Authoring an `apm-policy.yml` (warn→block) is Ch9.
- **Minimal example:** none authored in Ch1 (keys verified against the real CLI in Ch9). Grounded
  fact to cite: with no git remote, `apm policy status` reports `Outcome: no_git_remote` and
  `Effective rules: none` (scratch-dir output, v0.23.1).

### 4. `apm install` — restore
- **Implements concept:** **Portability + reproducibility** — the one command that turns a manifest
  (and lockfile) into a configured, identical agent context on any machine/harness.
- **Usage:** The daily consumer verb. From the docs, the whole restore is:
  ```bash
  git clone <repo> && cd <repo> && apm install
  # "Copilot, Claude, Cursor, OpenCode, Codex, Gemini, Windsurf, Kiro --
  #  every harness configured in one command" (official docs)
  ```
  Bare `apm install` reads `apm.yml`; `apm install --frozen` reproduces the lockfile exactly (CI).
- **When to use / when not:** Use bare `install` day-to-day; use `--frozen` in CI to fail on drift.
  It is **not** a runtime sandbox — it deploys/config files; runtime behavior stays the harness's
  domain. Restore is exercised for real in Ch5.
- **Minimal example (offline, verified):** on the empty scratch project,
  `apm install --dry-run` → `No dependencies found in apm.yml` … `no changes made` (exit 0). With
  nothing to deploy, no target is required.

### 5. Harness targets — parity across agents
- **Implements concept:** **Portability** — the "every harness" half of *portable by manifest*.
- **Usage:** A **target** is an agent harness APM can deploy the same primitives to. Resolution
  order (v0.23.1 `apm install --help`): **`--target` > `apm.yml` `targets:` > `apm config` target >
  auto-detect**. Auto-detect looks for a harness's marker files; declare `targets:` to pin
  explicitly. Inspect with `apm targets` (table) or `apm targets --json`.
- **When to use / when not:** Pin `targets:` in `apm.yml` when you want deterministic, team-wide
  parity (recommended for shared repos); rely on auto-detect for a solo machine that already has a
  harness configured. Pin explicitly if APM detects a harness you don't intend (e.g. a stray
  `CLAUDE.md` that is documentation, not config).
- **Minimal example (targets fragment):**
  ```yaml
  # apm.yml -- deploy the same context to two harnesses
  targets:
    - copilot
    - claude
  ```

## Caveats & surprises for the author

1. **Install requires a target — there is NO default copilot anymore.** This is the headline Ch1
   correction to common blog claims. Confirmed offline: adding a self-defined MCP server to the empty
   project with no active target neither errors nor silently deploys — APM **skips** and prints
   (v0.23.1, verbatim):
   > `[x] Skipping all MCP config writes -- could not resolve active targets.`
   > `[x] No harness detected`
   > `Previously APM defaulted to copilot; this is now explicit.`
   Nothing to deploy + no target = a no-op (exit 0). Something to deploy + no resolvable target =
   deployment is skipped until you pass `--target` or declare `targets:`. So "one `apm install`
   configures every harness" is true **only after** targets are resolved (pinned or auto-detected).
2. **`targets:` is emitted commented-out** by `apm init`, i.e. opt-in. And note: when the CLI
   rewrites `apm.yml` (e.g. after `--mcp`), the commented `targets:` block is **not preserved** —
   don't rely on the comment surviving edits.
3. **The three files don't all appear at once.** `apm init` writes only `apm.yml`.
   `apm.lock.yaml` appears on resolve/install; `apm-policy.yml` is an **org-remote** artifact
   discovered from git, not a local file you get by default (`no_git_remote` in a bare scratch dir).
4. **"Secure by default" is the fourth property in disguise.** Ch1 lists four *properties* but APM
   ships three *promises*; provenance/security is carried by content hashes in the lockfile plus
   install-time scanning (hidden-Unicode, transitive-MCP blocking). Name it in Ch1; prove it in Ch8.
5. **Not a runtime sandbox.** `apm install` configures files; it doesn't police what the agent does
   at runtime — worth stating early so readers don't over-claim.
